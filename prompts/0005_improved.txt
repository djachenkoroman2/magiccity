Ты senior Python-инженер и архитектор процедурных генераторов городских point clouds.

В репозитории уже есть MVP `citygen`: CLI-утилита на Python, YAML-конфиги, детерминированный `seed`, один тайл, регулярная grid-сетка дорог, тротуары, прямоугольные здания, surface sampling, ASCII PLY export и metadata JSON. Основная концепция описана в `prompts/0003.md`, первый MVP описан в `prompts/0003_mvp.md`, а реализация MVP была задана через `prompts/0004_improved.txt`.

Нужно реализовать следующий этап MVP: расширить генератор так, чтобы он поддерживал несколько моделей улично-дорожной сети, тайловую генерацию без швов, глобальные urban fields и 3-4 городских биома. Не останавливайся на описании подхода: внеси рабочие изменения в код, конфиги, документацию и тесты.

Главная цель: текущий `citygen` должен научиться генерировать разные типы городской структуры, а не только регулярную сетку. При этом старые конфиги должны продолжить работать.

## Обязательные Принципы

1. Сохрани обратную совместимость:
   - `configs/mvp.yaml` должен запускаться без изменений;
   - существующие `demo_*.yaml` должны либо продолжить работать, либо быть аккуратно обновлены с сохранением смысла;
   - `roads.model: grid` должен остаться поддержанным;
   - текущий CLI `uv run citygen --config ... --out ...` должен остаться рабочим.

2. Сохрани детерминированность:
   - одинаковый seed и одинаковый конфиг дают одинаковый результат;
   - генерация тайла не должна зависеть от порядка запуска соседних тайлов;
   - случайность получай через стабильные hash-based RNG от `seed`, координат, id объектов и namespace, а не через одно глобальное состояние.

3. Работай в мировых координатах:
   - road network, urban fields и biome map должны вычисляться от глобальных координат `x/z`;
   - тайл должен генерироваться с `margin_m`, а затем crop-иться до bbox тайла;
   - соседние тайлы с одним seed должны выглядеть согласованно на границах.

4. Не делай преждевременно полноценный GIS или LiDAR:
   - достаточно процедурной геометрии и surface sampling;
   - не добавляй тяжелые зависимости без необходимости;
   - не реализуй пока настоящие лучи LiDAR, occlusion model, intensity, LAS/LAZ, машины, деревья, ЛЭП и подстанции, если это мешает завершить этот этап.

## Новые Возможности

### 1. Несколько Моделей Уличной Сети

Расширь `roads.model`. Минимально поддержи:

```yaml
roads:
  model: grid
```

Уже есть и должен остаться.

```yaml
roads:
  model: radial_ring
```

Радиально-кольцевая система:
- есть городской центр;
- есть радиальные лучи от центра;
- есть кольцевые дороги на заданных радиусах или с заданным spacing;
- хорошо показывает downtown / old city структуру.

```yaml
roads:
  model: radial
```

Радиальная, или лучевая, система:
- дороги расходятся от центра;
- кольца не обязательны;
- можно добавить редкие поперечные связи, если это просто.

```yaml
roads:
  model: linear
```

Линейная система:
- город вытянут вдоль главной оси;
- дороги идут преимущественно вдоль этой оси;
- добавь поперечные улицы с меньшей частотой.

```yaml
roads:
  model: organic
```

Органическая система:
- дороги не обязаны быть прямой сеткой;
- направление может плавно меняться по noise / vector field;
- дороги должны адаптироваться к рельефу хотя бы эвристически: избегать слишком крутых участков или менять направление при высоком `terrain.height_noise_m`;
- достаточно приближенного surface mask / polyline подхода, не нужен идеальный graph solver.

```yaml
roads:
  model: mixed
```

Смешанная система:
- выбирает разные road pattern по biome или по urban fields;
- например центр `radial_ring`, planned residential `grid`, suburb `organic`, industrial `linear`;
- должна демонстрировать работу глобальных полей и биомов.

Дополнительно, если успеваешь, добавь:

```yaml
roads:
  model: free
```

Свободная / хаотичная система:
- менее регулярная сеть;
- можно реализовать как набор детерминированных полилиний, соединяющих локальные центры интереса, плюс случайные ответвления;
- не обязательно делать полноценный planar graph, но результат должен визуально отличаться от grid/organic.

Важно: не обязательно строить идеальный топологический граф дорог с кварталами промышленного уровня. Для этого этапа допустимо реализовать road mask на основе набора детерминированных road primitives: lines, radial rays, rings, noisy polylines. Но код должен быть расширяемым и понятным.

### 2. Road Primitives И Surface Classification

Сейчас `surface_kind()` фактически определяет road/sidewalk по расстоянию до осей grid. Нужно обобщить это:

- введи представление road network или road mask, которое работает для разных моделей;
- для каждой точки `(x, z)` можно определить:
  - расстояние до ближайшей road centerline;
  - является ли точка `road`;
  - является ли точка `sidewalk`;
  - является ли точка `ground`;
- road width и sidewalk width должны продолжать управляться YAML-конфигом;
- building placement не должен ставить здания на road/sidewalk.

Рекомендуемый минимальный подход:

```text
RoadPrimitive:
  - line segment / infinite grid line / radial ray / ring / sampled polyline
  - distance_to(x, z)
  - optional tangent / direction
```

Для MVP можно хранить road primitives в scene и использовать nearest distance для classification.

### 3. Глобальные Urban Fields

Добавь глобальные поля города, вычисляемые от мировых координат:

```text
centrality       // близость к городскому центру
density          // плотность застройки
height_potential // допустимая высотность
green_index      // степень озелененности / открытости
industrialness   // вероятность промышленного характера
orderliness      // регулярность планировки
```

Поля должны быть:

- детерминированными;
- плавными в мировых координатах;
- используемыми хотя бы в части решений генерации:
  - выбор road model для `mixed`;
  - выбор biome;
  - высота зданий;
  - плотность или вероятность зданий;
  - footprint / setback, если это удобно.

Добавь YAML-секцию, например:

```yaml
urban_fields:
  enabled: true
  center_x: 0
  center_z: 0
  city_radius_m: 1200
  noise_scale_m: 350
  density_bias: 0.0
  industrial_bias: 0.0
  green_bias: 0.0
```

Если секция отсутствует, используй defaults так, чтобы старые конфиги работали.

### 4. Городские Биомы

Добавь 3-4 биома минимум:

```text
downtown
residential
industrial
suburb
```

Можно добавить `park`, `old_town`, `waterfront`, если это не раздует задачу.

Биом должен определяться по urban fields. Примерная логика:

```text
if centrality high and density high:
    downtown
elif industrialness high:
    industrial
elif density medium:
    residential
else:
    suburb
```

Биомы должны реально влиять на генерацию:

- `downtown`: выше здания, выше плотность, более регулярные или радиально-кольцевые дороги;
- `residential`: средняя высота, средняя плотность, grid/organic mix;
- `industrial`: крупные footprint, низкие/средние здания, linear или sparse roads;
- `suburb`: низкая высота, редкие здания, больше открытой земли, organic или sparse grid.

Добавь metadata:

- используемые biome names;
- распределение точек или sampled cells по biome, если это просто;
- config с urban fields и biome settings.

### 5. Тайлы И Batch Generation

Текущий конфиг описывает один тайл:

```yaml
tile:
  x: 0
  z: 0
  size_m: 256
  margin_m: 32
```

Сохрани этот формат.

Дополнительно поддержи генерацию нескольких тайлов из одного конфига. Возможный формат:

```yaml
tiles:
  items:
    - {x: 0, z: 0}
    - {x: 1, z: 0}
    - {x: 0, z: 1}
    - {x: 1, z: 1}
  size_m: 256
  margin_m: 48
```

Или:

```yaml
tiles:
  x_range: [0, 2]
  z_range: [0, 2]
  size_m: 256
  margin_m: 48
```

Выбери один простой формат или поддержи оба, если это недорого. Важно:

- старый `tile` остается рабочим;
- `tiles` позволяет получить несколько `.ply`;
- если `--out` указывает директорию, складывай туда `tile_X_Z.ply`;
- если `--out` указывает конкретный `.ply`, а в конфиге несколько тайлов, выдай понятную ошибку или используй безопасную стратегию именования. Лучше понятная ошибка.

## Тесты

Добавь или обнови тесты так, чтобы проверять:

1. Старый `configs/mvp.yaml` или минимальный old-style config все еще валиден.
2. Все новые `roads.model` проходят validation.
3. Неизвестный `roads.model` дает понятную ошибку.
4. Одинаковый seed и конфиг дают одинаковый результат.
5. Разный seed меняет road/building/point result хотя бы для одной новой модели.
6. Для каждого road model есть ненулевые классы `road` и `sidewalk`.
7. При включенных buildings есть ненулевые `building_facade` и `building_roof`, если конфиг это подразумевает.
8. `mixed` действительно создает больше одного biome в metadata или sampled distribution.
9. Multi-tile config создает несколько PLY и metadata файлов при output directory.

Запусти проверки:

```bash
uv run python -m unittest discover -s tests
```

Если в окружении нет `uv`, используй доступный локальный `.venv/bin/python -m unittest discover -s tests` и явно напиши, чем проверял.

## Критерии Приемки

Работа считается завершенной, если:

- старый MVP-конфиг запускается;
- новые demo configs запускаются;
- минимум `grid`, `radial_ring`, `radial`, `linear`, `organic`, `mixed` реально поддержаны;
- разные road models дают различимое распределение road/sidewalk geometry;
- urban fields вычисляются в мировых координатах и влияют на generation decisions;
- минимум 4 биома существуют и влияют на здания или дороги;
- multi-tile generation работает из одного конфига;
- PLY и metadata создаются для single-tile и multi-tile modes;
- README обновлен;
- тесты проходят.
