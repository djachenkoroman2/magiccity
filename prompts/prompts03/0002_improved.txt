Ты senior Python-инженер, архитектор процедурной генерации и technical writer для проекта MagicCity.

В репозитории есть рабочий MVP `citygen`: CLI на Python, YAML-конфиги, детерминированный `seed`, тайлы, процедурный рельеф, road primitives, несколько road models, road profiles, worldgen catalogs, urban fields, биомы, parcels, fences, trees, mobile LiDAR, разные building footprints/roofs, surface sampling, ASCII PLY export, metadata JSON и русскоязычная документация в `doc/` с HTML-сборкой через MkDocs. Координатная конвенция проекта: горизонтальная плоскость `x/y`, высота `z`.

Нужно реализовать новый опциональный слой generated objects: транспортные средства. Не ограничивайся описанием подхода: внеси рабочие изменения в код, конфиги, тесты и документацию.

## Главная цель

Добавить в `citygen` независимую подсистему генерации транспорта, которая:

- размещает транспорт только там, где он физически может находиться: на проезжей части, парковочных/дворовых площадках или промышленных площадках;
- поддерживает разные форм-факторы: легковые автомобили, грузовики, автобусы, спецтехнику и тракторы;
- параметризуется через YAML;
- учитывает road profiles, направление дороги, биомы, parcels/buildings/fences/trees и рельеф;
- участвует в surface sampling и mobile LiDAR;
- экспортирует semantic class/RGB и metadata;
- интегрирована в worldgen pipeline как отдельный понятный этап;
- отражена в Markdown- и HTML-документации.

## Функциональные требования

1. Добавить новую секцию конфигурации `vehicles`.

   Минимальные ожидаемые параметры:

   - `enabled`: включает или выключает слой транспорта, по умолчанию `false`;
   - `density_per_km` или аналогичный параметр плотности транспорта вдоль дорог;
   - `parking_density_per_ha` или аналогичный параметр для стоящих машин на площадках/внутри parcels;
   - `min_spacing_m`: минимальная дистанция между транспортными средствами;
   - `placement_modes`: например `road`, `parking`, `industrial_yard`, `mixed`;
   - `vehicle_type`: конкретный тип или `mixed`;
   - `weights` для выбора типа транспорта при `vehicle_type: mixed`;
   - `biome_density_multipliers` или аналогичный механизм для разных биомов;
   - размеры по типам: `length_m`, `width_m`, `height_m`, `wheel_radius_m`, `clearance_m` или catalog defaults;
   - `orientation_jitter_degrees`;
   - clearances от buildings, fences, trees, границ тайла, тротуаров и road medians;
   - параметры road placement: разрешенные road profiles, lane offset, parked/traffic ratio, side-of-road placement;
   - `sample_spacing_m` для плотности точек транспорта.

   Поддержи строгую валидацию секции `vehicles`: неизвестные поля должны давать понятную ошибку.

2. Поддержать форм-факторы транспорта.

   Минимальный набор:

   - `car`: легковой автомобиль;
   - `truck`: грузовик или фургон;
   - `bus`: городской/пригородный автобус;
   - `emergency`: спецтехника, например пожарная/служебная машина;
   - `tractor`: трактор или сельхоз/utility vehicle.

   Для каждого типа должны быть catalog defaults: габариты, цветовая палитра, высота кузова, колесная база, допустимые placement modes и теги. Если вводятся aliases (`sedan`, `van`, `lorry`, `firetruck`, `utility`, `farm_tractor`), опиши их в catalogs/docs и покрой тестами.

3. Размещение транспорта должно учитывать поверхность и занятость сцены.

   Транспорт можно размещать:

   - на `road`/carriageway;
   - на parking/yard surfaces, если они реализуются как отдельный слой или как допустимые зоны внутри parcels;
   - на industrial parcels/yards для грузовиков, спецтехники и тракторов;
   - на residential/suburb parcels для parked cars, если это не конфликтует с buildings/fences/trees;
   - с ориентацией вдоль road primitive или локальной оси площадки;
   - с `z`, вычисленным через `terrain_height` в точке установки.

   Транспорт нельзя размещать:

   - на `ground`, если это не явно созданная parking/yard площадка;
   - на `sidewalk`;
   - на `road_median`, если не добавлен явный параметр;
   - внутри building footprints;
   - в зоне clearance вокруг зданий;
   - на fences/fence foundations;
   - внутри кроны/ствола дерева или слишком близко к деревьям;
   - за пределами crop bbox тайла;
   - на слишком узкой дороге, если габариты типа транспорта не помещаются в carriageway.

   Алгоритм размещения должен быть детерминированным от `seed`, координат тайла, road primitive/parcel id, биома и локального id транспорта.

4. Плотность и типы должны зависеть от биома.

   Сделай реалистичные значения по умолчанию:

   - `downtown`: больше автобусов, такси/легковых, служебной техники; высокая плотность на дорогах, мало тракторов;
   - `residential`: средняя плотность легковых, немного грузовиков/служебных;
   - `suburb`: больше parked cars и utility vehicles, низкая плотность движения;
   - `industrial`: больше грузовиков, спецтехники и тракторов/utility, меньше легковых;
   - должна быть возможность явно задать `0` для типа или биома.

5. Архитектура должна быть модульной.

   Добавь независимый модуль генерации, например `citygen/vehicles.py`, не раздувая `generator.py`, `sampling.py` и `mobile_lidar.py` сверх необходимости.

   Ожидаемые сущности:

   - dataclass для экземпляра транспорта;
   - dataclass/объект для спецификации типа транспорта;
   - функция генерации транспорта по scene context;
   - helpers для road/parcel/parking placement;
   - функция семплирования корпуса, колес и окон;
   - helpers для пересечений mobile LiDAR.

   Новая подсистема должна быть встроена в существующий pipeline, но не должна менять старые сцены при `vehicles.enabled: false`.

6. Интегрировать транспорт в пайплайн.

   Добавь отдельный stage, например `vehicles`, после roads/parcels/buildings/fences/trees и до sampling, если это соответствует текущей архитектуре.

   Обнови:

   - worldgen catalogs/stages;
   - CLI/progress diagnostics;
   - metadata pipeline summary;
   - документацию с перечнем стадий всего pipeline;
   - tests, которые проверяют наличие стадии в docs и metadata.

   Цель: пользователь должен видеть, что генерация транспорта является отдельным этапом, а не скрытой частью sampling.

7. Семантические классы и RGB.

   Добавь новые классы для транспорта, не меняя существующие class ids:

   - `vehicle_body`;
   - `vehicle_wheel`;
   - `vehicle_window`;

   Если считаешь полезным, добавь отдельные классы `vehicle_light` или `vehicle_load`, но не усложняй MVP без необходимости. Назначь стабильные новые ids и RGB-цвета. Обнови class mapping, RGB metadata, docs и тесты. Существующие semantic class ids менять нельзя.

8. Геометрия транспорта.

   MVP может использовать аналитические формы, а не полноценные mesh assets:

   - корпус как oriented box с небольшими деталями;
   - кабина/верхняя часть для truck/bus/tractor;
   - колеса как цилиндры или упрощенные диски;
   - окна как отдельные sampled patches на корпусе;
   - optional roof/hood slopes, если это легко встроить.

   Геометрия должна хранить orientation в world-space и корректно работать с координатной конвенцией `x/y` + `z`.

9. Sampling.

   Surface sampling должен генерировать точки:

   - на корпусе;
   - на колесах;
   - на окнах;
   - с корректным `z`, основанным на рельефе в точке установки;
   - с semantic class и RGB;
   - с контролируемой плотностью через `vehicles.sample_spacing_m`.

   Не допускай чрезмерного роста количества точек на больших тайлах: используй разумные defaults, ограничения и sampling caps на объект, если нужно.

10. Mobile LiDAR.

    Если mobile LiDAR включен, лучи должны уметь попадать в транспорт:

    - корпус должен работать как oriented box obstacle;
    - колеса/окна могут быть либо отдельными пересечениями, либо частью упрощенного корпуса, если это явно описано;
    - ближайшее попадание по лучу должно учитывать транспорт вместе с terrain, roads, buildings, fences и trees;
    - при включенных occlusions транспорт может закрывать объекты позади.

    Если полная физика oriented boxes/колес слишком велика для одного изменения, реализуй минимальный корректный вариант и явно зафиксируй ограничение в документации.

11. Metadata.

    Добавь диагностические поля:

    - общее число транспортных средств;
    - counts по типу (`car`, `truck`, `bus`, `emergency`, `tractor`);
    - counts по placement mode (`road`, `parking`, `industrial_yard`);
    - counts по биому;
    - средние/минимальные/максимальные габариты;
    - количество body/wheel/window points;
    - параметры `vehicles` после применения defaults;
    - участие транспорта в `object_feature_counts`;
    - supported vehicle types и aliases/catalog summary.

12. Demo-конфиги.

    Добавь или обнови demo-конфиг:

    - предпочтительно отдельный `configs/demo_vehicles.yaml`;
    - либо аккуратное включение транспорта в `configs/demo_universal_showcase.yaml`, если это не делает showcase слишком тяжелым.

    В демо должны быть видны разные типы транспорта и разные placement modes. Если parking/yard surfaces реализованы минимально, продемонстрируй хотя бы road vehicles + parked vehicles внутри подходящих parcels.

13. Документация.

    Обнови Markdown-документацию на русском языке:

    - `doc/configuration_reference.md`;
    - отдельный тематический документ, например `doc/vehicles.md`;
    - `doc/generated_objects.md`;
    - `doc/sampling.md`;
    - `doc/worldgen_catalogs.md`;
    - `doc/universal_showcase.md`, если showcase изменен;
    - `README.md` и `configs/README.md`, если добавлен новый demo.

    Обнови `mkdocs.yml` и пересобери HTML:

    ```bash
    uv run mkdocs build --clean
    ```

    В результате `doc/html/` должен содержать актуальную HTML-документацию, включая страницу про транспорт.

## Тесты

Добавь тесты, которые проверяют:

- загрузку и defaults секции `vehicles`;
- строгую валидацию неизвестных полей;
- валидацию отрицательных плотностей, размеров, spacing, clearances и неизвестных типов транспорта;
- aliases типов транспорта;
- `vehicles.enabled: false` не меняет старое поведение;
- при включенном транспорте появляются экземпляры транспорта и точки `vehicle_body`/`vehicle_wheel`/`vehicle_window`;
- транспорт не размещается на sidewalk, road_median без разрешения, buildings, fences и trees;
- road vehicles ориентированы вдоль road primitive или допустимого road direction;
- biome multipliers и weights реально влияют на количество/типы транспорта;
- одинаковый `seed` дает одинаковый транспорт и точки;
- metadata содержит vehicle counts, class mapping/RGB и object feature counts;
- mobile LiDAR, если включен, может попадать в транспорт или корректно документирует временное ограничение.

Минимально запусти:

```bash
uv run python -m unittest discover -s tests
uv run mkdocs build --clean
git diff --check
```

Если генерация полного showcase тяжелая, используй маленький тестовый конфиг или временный output в `/tmp`, но проверь, что новый layer действительно работает.

## Ограничения и совместимость

- Не меняй существующие semantic class ids.
- Не ломай существующие YAML-конфиги.
- Не делай транспорт включенным по умолчанию, если это меняет существующие outputs.
- Не размещай транспорт на natural ground без явно созданной parking/yard зоны.
- Не размещай транспорт на тротуарах и medians без явного параметра.
- Не добавляй внешние тяжелые зависимости без необходимости.
- Не подменяй реализацию только документацией: новый слой должен реально генерировать объекты и точки.
- Если добавляешь parking/yard surfaces как MVP-аппроксимацию, держи ее простой, документированной и покрытой тестами.

## Ожидаемый результат

Итоговая реализация должна добавить реалистичный, конфигурируемый и детерминированный слой транспорта: разные форм-факторы, biome-aware density, корректное размещение на дорогах/площадках, semantic classes/RGB, sampling, mobile LiDAR, metadata, pipeline stage, demo-конфиг, тесты и актуальную Markdown/HTML-документацию.
