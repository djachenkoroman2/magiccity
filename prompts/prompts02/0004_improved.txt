Необходимо улучшить CLI/runtime-вывод генератора, потому что генерация может выполняться долго и сейчас пользователю недостаточно понятно, что именно происходит, сколько стадий уже выполнено и какой результат получился.

Требуется добавить информативный, но не шумный вывод в консоль до запуска, во время выполнения pipeline и после завершения генерации.

Требования к функциональности:

1. Перед началом генерации вывести краткую сводку конфигурации:
   - путь к используемому config-файлу;
   - `seed`;
   - режим одного тайла или multi-tile;
   - координаты и размер тайла или список тайлов;
   - основные включенные подсистемы: roads, buildings, parcels, fences, mobile LiDAR, sampling, output;
   - ключевые параметры, влияющие на длительность выполнения: размер тайла, margin, spacing sampling, количество тайлов, mobile LiDAR ray density;
   - путь, куда будет записан результат.

2. Во время генерации выводить этапы pipeline в понятном порядке:
   - loading config;
   - resolving catalogs/worldgen context;
   - roads;
   - parcels;
   - buildings/objects;
   - fences;
   - sampling;
   - mobile LiDAR, если включен;
   - writing PLY;
   - writing metadata.

3. Для каждого этапа необходимо показывать прогресс:
   - название текущей стадии;
   - номер стадии и общее количество стадий;
   - статус `started` / `done`;
   - длительность стадии после завершения;
   - для multi-tile режима дополнительно показывать номер текущего тайла и общее количество тайлов.

4. Если на стадии можно получить содержательную промежуточную статистику, нужно вывести ее:
   - количество road primitives;
   - количество blocks/parcels;
   - количество зданий;
   - количество fence segments;
   - количество surface points;
   - количество LiDAR points;
   - количество итоговых points после crop/output mode;
   - при наличии mobile LiDAR: количество sensor positions, emitted rays, successful hits, missed/dropped/attenuated rays.

5. После завершения генерации вывести итоговый summary:
   - общий статус выполнения;
   - общее время генерации;
   - путь к PLY-файлу;
   - путь к metadata-файлу;
   - итоговое количество точек;
   - распределение точек по semantic class;
   - режим источников точек: surface only, additive или lidar only;
   - основные характеристики сцены: road models, building counts, parcel counts, fence counts, mobile LiDAR statistics, если соответствующие подсистемы включены.

6. Summary должен быть полезен как для одного тайла, так и для multi-tile запуска:
   - для каждого тайла выводить отдельный краткий результат;
   - после всех тайлов выводить общий итог по количеству файлов, точек и времени;
   - ошибки по одному тайлу должны быть понятными и содержать координаты тайла, если они доступны.

7. Вывод должен быть аккуратно структурирован:
   - не печатать огромные JSON-объекты целиком;
   - длинные mappings сокращать до наиболее важных значений;
   - использовать стабильный формат строк, пригодный для чтения человеком и для smoke-тестов;
   - не дублировать одну и ту же информацию слишком часто;
   - не ломать существующий stdout-контракт без необходимости.

8. Нужно предусмотреть настройку уровня подробности:
   - обычный режим по умолчанию должен показывать preflight, progress stages и final summary;
   - quiet-режим должен оставлять минимум вывода или только финальные пути;
   - verbose-режим должен показывать расширенную статистику по стадиям;
   - если в проекте уже есть CLI-паттерн для флагов, использовать его;
   - если такого паттерна нет, добавить понятные CLI-флаги, например `--quiet` и `--verbose`.

9. Необходимо учитывать ошибки:
   - если конфиг не загрузился или не прошел валидацию, вывести понятное сообщение с путем к config;
   - если генерация упала на стадии pipeline, указать стадию и тайл;
   - если запись PLY или metadata не удалась, указать целевой путь;
   - при ошибках не скрывать исходное исключение там, где оно полезно для разработчика.

10. Реализация должна опираться на реальные стадии проекта:
    - использовать существующие функции `load_config`, `generate_scene`, `sample_scene`, `write_ply`, `write_metadata` или их текущие аналоги;
    - не придумывать стадии, которых нет в runtime, без явной причины;
    - если для более точного прогресса нужно разнести крупные функции на подэтапы, сделать это аккуратно и без изменения результата генерации;
    - сохранить детерминированность: добавление логирования не должно менять seed, RNG, порядок генерации точек или содержимое PLY/metadata.

11. Нужно добавить или обновить тесты:
    - CLI выводит preflight-сводку перед генерацией;
    - CLI выводит стадии pipeline и финальный summary;
    - quiet/verbose режимы работают ожидаемо;
    - multi-tile запуск выводит прогресс по каждому тайлу;
    - итоговые PLY и metadata по-прежнему создаются;
    - добавление progress output не меняет детерминированный результат генерации.

12. Необходимо обновить документацию:
    - описать новые CLI-флаги;
    - показать пример обычного вывода;
    - показать пример quiet/verbose использования;
    - указать, где смотреть итоговые PLY и metadata;
    - объяснить, какие метрики появляются в финальном summary.

Итоговая реализация должна сделать долгий запуск генератора наблюдаемым: пользователь заранее видит, какой config и какие тайлы будут обработаны, во время выполнения понимает текущую стадию и прогресс, а после завершения получает компактную статистику результата, пути к файлам и основные характеристики сгенерированного облака точек.
