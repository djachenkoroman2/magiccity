Необходимо реализовать детальный прогресс выполнения для стадии `sampling`, потому что именно этот этап может занимать заметное время при больших тайлах, высокой плотности точек, большом количестве зданий, ограждений или включенном mobile LiDAR.

Требуется добавить отображение прогресса внутри шага `sampling` при запуске проекта через CLI. Можно использовать `tqdm`, если он хорошо вписывается в проект, но решение не должно ломать существующий вывод, тесты и работу в quiet/verbose режимах.

Требования к функциональности:

1. Прогресс должен отображаться именно внутри стадии `sampling`, а не только на уровне общей стадии pipeline:
   - sampling tile surfaces: ground, road, sidewalk, road_median;
   - sampling buildings: roofs and facades;
   - sampling fences, если они включены;
   - mobile LiDAR ray sampling, если `mobile_lidar.enabled: true`.

2. Для surface sampling нужно показывать понятный прогресс:
   - количество обработанных grid samples или строк grid;
   - общий объем работы, если его можно вычислить заранее;
   - количество уже созданных surface points;
   - отдельные counters для ground/hardscape/building/fence points, если это можно сделать без больших затрат.

3. Для building sampling нужно учитывать:
   - общее количество зданий;
   - номер текущего здания;
   - количество roof/facade points после обработки здания;
   - отсутствие зданий должно корректно отображаться как быстрый завершенный подэтап.

4. Для fence sampling нужно учитывать:
   - количество fence segments;
   - номер текущего segment;
   - количество `fence` и `fence_foundation` points;
   - если fences выключены или сегментов нет, подэтап не должен создавать шумный вывод.

5. Для mobile LiDAR нужно показывать прогресс по лучам:
   - количество sensor positions;
   - количество emitted rays;
   - количество обработанных rays;
   - количество successful hits;
   - dropped/missed/attenuated rays;
   - итоговое количество LiDAR points.

6. Нужно выбрать аккуратную стратегию вывода:
   - в обычном режиме показывать компактный progress, который не засоряет консоль;
   - в `--quiet` не показывать progress bars и промежуточные строки;
   - в `--verbose` показывать более подробные counters по подэтапам;
   - если stdout не является TTY, избегать динамических progress bars и печатать стабильные строки, пригодные для тестов и логов;
   - если используется `tqdm`, он должен быть опциональным или добавлен в зависимости проекта осознанно.

7. Progress не должен менять результат генерации:
   - не менять seed;
   - не менять порядок RNG calls;
   - не менять порядок точек в итоговом PLY;
   - не менять class ids, RGB и metadata;
   - не добавлять недетерминированные данные в PLY/metadata.

8. Нужно аккуратно встроиться в текущую архитектуру:
   - использовать существующий CLI progress/reporting механизм, если он уже есть;
   - если есть callback для стадий pipeline, расширить его под progress внутри `sampling`;
   - не смешивать бизнес-логику генерации с форматированием CLI;
   - функции sampling должны оставаться пригодными для тестов и вызова из Python без CLI.

9. Нужно предусмотреть multi-tile режим:
   - progress должен указывать текущий tile;
   - прогресс каждого тайла должен начинаться и завершаться отдельно;
   - итоговый summary по всем тайлам должен остаться читаемым.

10. Нужно добавить тесты:
    - обычный CLI-режим выводит progress для sampling;
    - `--quiet` скрывает progress;
    - `--verbose` выводит расширенные counters;
    - при включенном mobile LiDAR появляются ray progress/statistics;
    - multi-tile запуск показывает sampling progress по каждому tile;
    - PLY, metadata и детерминированность результата не меняются из-за progress output.

11. Нужно обновить документацию:
    - описать, что стадия `sampling` теперь показывает внутренний progress;
    - объяснить отличие обычного, quiet и verbose режимов;
    - показать короткий пример вывода;
    - указать, что progress является диагностическим выводом и не влияет на результат генерации.

Итоговая реализация должна сделать длительную стадию `sampling` наблюдаемой: пользователь должен видеть, что генератор не завис, понимать, какая часть sampling сейчас выполняется, сколько точек или лучей уже обработано, и получать финальные counters без изменения детерминированного результата.
