Ты senior Python-инженер и архитектор утилит командной строки для процедурной генерации 3D point clouds.

Нужно реализовать MVP генератора синтетической городской застройки в виде облака точек. Основная концепция описана в `prompts/0003.md`, а первый шаг MVP описан в `prompts/0003_mvp.md`. Используй эти файлы как продуктовый и архитектурный контекст.

Цель: сделать рабочую CLI-утилиту на Python, запускаемую через локальное окружение `uv`, где все параметры генерации задаются через YAML-конфиг. MVP должен генерировать один или несколько городских тайлов, создавать простые дороги, прямоугольные здания, базовые классы точек и экспортировать результат в файл.

Не останавливайся на описании подхода: реализуй работающий прототип в репозитории.

Обязательные требования:

1. Окружение и зависимости
   - используй `uv` для управления окружением и запуска;
   - если в репозитории нет проекта Python, создай минимальный `pyproject.toml`;
   - добавь только необходимые зависимости;
   - для YAML используй стандартную надежную библиотеку, например `PyYAML`;
   - для численных операций можно использовать `numpy`;
   - избегай тяжелых зависимостей, если MVP можно сделать проще.

2. CLI
   - сделай команду запуска через `uv run ...`;
   - CLI должен принимать путь к YAML-конфигу;
   - CLI должен позволять указать output path или output directory;
   - добавь `--help`;
   - при ошибках конфига выводи понятные сообщения.

   Пример желаемого запуска:

   ```bash
   uv run citygen --config configs/mvp.yaml --out outputs/tile_0_0.ply
   ```

   Если entrypoint `citygen` неудобен для текущей структуры проекта, можно использовать:

   ```bash
   uv run python -m citygen --config configs/mvp.yaml --out outputs/tile_0_0.ply
   ```

3. YAML-конфиг
   - создай пример конфига, например `configs/mvp.yaml`;
   - все важные параметры генерации должны приходить из YAML;
   - задай значения по умолчанию там, где это уместно;
   - провалидируй обязательные поля.

   Минимально поддержи параметры:

   ```yaml
   seed: 42
   tile:
     x: 0
     y: 0
     size_m: 256
     margin_m: 32
   terrain:
     base_height_m: 0
     height_noise_m: 1.5
   roads:
     model: grid
     spacing_m: 64
     width_m: 10
     sidewalk_width_m: 3
   buildings:
     enabled: true
     min_height_m: 8
     max_height_m: 60
     setback_m: 6
     footprint_min_m: 12
     footprint_max_m: 36
   sampling:
     mode: surface
     ground_spacing_m: 2.0
     road_spacing_m: 1.5
     building_spacing_m: 2.0
   output:
     format: ply
     include_rgb: true
     include_class: true
   ```

4. MVP-геометрия
   - реализуй генерацию одного тайла в мировых координатах;
   - используй `seed` так, чтобы повторный запуск с тем же конфигом давал одинаковый результат;
   - сделай простую регулярную сетку дорог;
   - выдели дорожное полотно и тротуары;
   - разбей пространство между дорогами на кварталы;
   - размести простые прямоугольные здания внутри кварталов;
   - здания должны иметь высоту, footprint, крышу и фасады;
   - не допускай пересечения зданий с дорогами;
   - используй небольшой height/noise для поверхности земли, если это не усложняет MVP.

5. Классы точек и цвета
   - каждая точка должна иметь координаты `x, y, z`;
   - добавь RGB-цвета;
   - добавь semantic class хотя бы для:
     - `ground`;
     - `road`;
     - `sidewalk`;
     - `building_facade`;
     - `building_roof`.
   - классы можно хранить числом, но добавь явную таблицу соответствий в коде или metadata.

6. Семплирование point cloud
   - для MVP достаточно surface sampling, без полноценной LiDAR-симуляции;
   - земля, дороги и крыши могут семплироваться регулярной сеткой;
   - фасады зданий должны тоже иметь точки, а не только крыши;
   - плотность/spacing должны настраиваться через YAML;
   - добавь небольшой детерминированный jitter, чтобы облако не выглядело слишком искусственно.

7. Экспорт
   - обязательно реализуй экспорт в ASCII PLY;
   - PLY должен открываться в стандартных просмотрщиках point cloud/mesh;
   - если добавляешь class, сделай его отдельным свойством вершины;
   - создай output directory автоматически, если он не существует;
   - сохрани рядом metadata JSON с seed, конфигом, bbox, количеством точек и распределением классов.

8. Структура проекта
   - предложи и реализуй простую читаемую структуру модулей;
   - отдели CLI, чтение конфига, генерацию геометрии, семплирование и экспорт;
   - не смешивай всю логику в одном огромном файле, если это легко избежать.

   Возможная структура:

   ```text
   citygen/
     __init__.py
     __main__.py
     cli.py
     config.py
     generator.py
     geometry.py
     sampling.py
     export.py
     classes.py
   configs/
     mvp.yaml
   outputs/
     .gitkeep
   tests/
     test_config.py
     test_determinism.py
     test_export.py
   ```

9. Проверка и тесты
   - добавь минимальные тесты, если в проекте есть тестовая инфраструктура или ее можно быстро добавить;
   - обязательно проверь детерминированность: одинаковый seed и конфиг дают одинаковое количество точек и одинаковые первые N точек;
   - проверь, что PLY создается и содержит ожидаемый header;
   - после реализации запусти доступные проверки через `uv run ...`;
   - если какую-то проверку запустить не удалось, явно объясни причину.

10. Качество реализации
    - код должен быть понятным и расширяемым;
    - добавь type hints там, где они помогают;
    - избегай скрытого глобального состояния;
    - случайность должна быть завязана на seed и координаты, а не на недетерминированное состояние;
    - не реализуй пока сложную LiDAR-физику, деревья, машины, ЛЭП и подстанции, если это мешает быстро получить MVP.

Ожидаемый результат работы:

- создана CLI-утилита генерации городского point cloud;
- есть пример YAML-конфига;
- есть команда запуска;
- есть сгенерированный пример PLY или понятная инструкция, как его получить;
- есть metadata JSON;
- есть краткое описание, что реализовано и как расширять дальше.

Критерии приемки:

- `uv run citygen --config configs/mvp.yaml --out outputs/mvp_tile.ply` или эквивалентная команда успешно выполняется;
- output PLY содержит точки земли, дорог, тротуаров, фасадов и крыш;
- повторный запуск с тем же seed дает тот же результат;
- изменение seed меняет городскую сцену;
- параметры из YAML реально влияют на генерацию;
- проект можно продолжать развивать в сторону следующих этапов из `prompts/0003.md`.

Перед началом реализации:

1. Осмотри текущую структуру репозитория.
2. Проверь наличие `pyproject.toml`, `uv.lock`, существующих Python-модулей и тестов.
3. Кратко сформулируй план изменений.
4. Затем реализуй MVP, запусти проверки и в конце дай короткий отчет с командами запуска.
